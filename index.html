Sobre mi

¡Hola! Soy Francisco.

Mi camino en la tecnología comenzó con la Ingeniería Electrónica, donde aprendí a entender cómo funcionan las cosas desde la base. Esa curiosidad me llevó a convertirme en SysAdmin y luego en Cloud Engineer, gestionando servidores físicos y virtuales en AWS y GCP, asegurando que los sistemas nunca fallen.

Pero quería estar más cerca del usuario. Así descubrí el Frontend. Hoy disfruto transformando diseños en código real (React/SCSS), aplicando mis conocimientos de infraestructura para que cada sitio vuele en términos de velocidad.

Finalmente, mi obsesión por la precisión me llevó al mundo del QA. No solo pruebo aplicaciones manualmente en web y móvil con Android Studio; automatizo el futuro con Playwright. Al conocer el código desde adentro, optimizo las pruebas colocando test-ids estratégicos, logrando que la automatización sea realmente eficiente.

En resumen: Construyo, despliego y aseguro la calidad.

Experiencia Profesional como Quality Asurance

QA Automation | Framework de Automatización E2E con Playwright & React

2022 - 2025

Willdom LLC, USA, FL

Mi Enfoque Único: "Testability by Design"
  • A diferencia de un QA tradicional, utilicé mi perfil de Frontend Developer para intervenir el código fuente:
  • Implementación de data-testid: En lugar de usar selectores frágiles como .btn-primary o XPaths complejos, integré atributos personalizados en los componentes de React. Esto garantiza que, aunque el diseño cambie, los tests sigan funcionando.
  • Arquitectura Page Object Model (POM): Estructuré el código en capas para separar la lógica de los tests de la interacción con la interfaz, facilitando la escalabilidad del proyecto.
Stack Tecnológico
  • Core: Playwright (TypeScript).
  • Frontend: React / SCSS (Intervención de componentes).
  • Herramientas: Postman (Validación de APIs previo a E2E), GitHub Actions.
  • Reportes: Allure Reports con evidencias en video y capturas de pantalla automáticas en caso de fallo.
Resultados y Valor Agregado
  • Mantenibilidad: Reducción del tiempo dedicado a reparar tests rotos por cambios en la UI.
  • Integración Continua: Configuración de un pipeline en GitHub Actions que bloquea despliegues si la suite crítica de regresión no es exitosa.
